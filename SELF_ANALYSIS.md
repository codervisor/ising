## Signals Detected

### 1. Ghost Coupling (severity: 1.00) — HIGH

```
ising-builders/src/change.rs <-> ising-builders/src/structural.rs
```

**What this means:** These two files change together frequently (100% temporal coupling) but have **no structural dependency** between them. They don't import each other, call each other's functions, or share types directly — yet every time one changes, the other does too.

**Why it matters:** Ghost coupling is the signature of a hidden architectural relationship. In this case, `change.rs` (git history analyzer) and `structural.rs` (AST parser) are the two halves of the graph-building pipeline. They evolve in lockstep because changes to one layer's schema or node format require matching updates in the other. This suggests a shared abstraction (perhaps a `BuilderTrait` or shared node schema) could make the coupling explicit and reduce coordination overhead.

### 2. Stable Core (severity: 0.10) — GUARD

```
ising-core/src/ignore.rs
```

**What this means:** This module has high structural importance but low change frequency and zero defects. It is a **stable foundation** the rest of the system relies on.

**Why it matters:** `ignore.rs` implements `.isingignore` glob matching — a well-defined, self-contained concern. Its stability is a positive signal. The GUARD classification means: protect this module from unnecessary changes, as modifications here could ripple outward.

## Architecture Observations

1. **Zero cycles** — The dependency graph is a clean DAG. The crate structure (`core` -> `builders` -> `analysis` -> `db` -> `cli`) enforces layering at compile time.

2. **Ghost coupling as self-diagnosis** — The most interesting finding is that Ising detected its own architectural smell: the implicit coupling between its two builder modules. This validates the tool's signal detection on a real (if small) codebase.

3. **Complexity concentration** — 76% of total complexity lives in just 3 files (`structural.rs`, `db/lib.rs`, `cli/main.rs`). This is typical for a young codebase but flags future refactoring targets.

4. **Stable core** — `ising-core` modules (`graph.rs`, `config.rs`, `ignore.rs`, `metrics.rs`) are all low-churn, low-hotspot. The foundational layer is solid.
